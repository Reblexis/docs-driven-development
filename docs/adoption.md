# Adoption

Adoption is itself docs-first, and it is the same move everywhere: write the root doc — what this thing is and what it must do — and let agents make reality match it.

A new project starts as a single doc and grows code underneath it. An existing project is treated as an implementation whose docs don't exist yet: describe what it should be, then govern it from those docs. Existing code neither blocks adoption nor gets special status — from the moment the docs exist, they are the source of truth.

How deep the doc hierarchy grows, how strictly conformance is checked, and how fast the codebase is brought into line are choices each project makes for itself, in its own docs. This methodology deliberately does not prescribe them.
